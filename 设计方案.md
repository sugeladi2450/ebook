# React 重构设计方案

## 1. 设计目标

本次重构不是重新设计一套新界面，而是在尽量保持原有 HTML 页面结构、CSS 类名、页面文案和视觉风格不变的前提下，把项目升级为基于 React 和 React Router 的前端工程。核心目标如下：

1. 用户启动项目后，可以在主页、书籍详情页、购物车页、订单页、个人信息页和登录页之间进行路由跳转。
2. 点击主页中的某一本书后，详情页展示的必须是该书对应的数据，页面之间通过 `props` 和路由 `state` 传递数据。
3. 页面中的展示数据统一放入前端数据文件中，由 React 工程读取，不再把数据散落在多个 HTML 和内联脚本里。
4. 工程结构体现 React 构件化思想，将重复结构拆分为可复用函数组件。

## 2. 保持原页面不变的做法

为了尽量贴近原始页面，本次重构采用了“原页面结构翻译成 JSX”的策略，而不是重新设计布局。

具体做法如下：

1. 保留原有 `css/style.css`、`css/books.css`、`css/book-detail.css`、`css/cart.css`、`css/orders.css`、`css/profile.css`、`css/login.css`。
2. React 页面组件中的 HTML 结构、类名命名、标题文案、按钮文案、分区顺序都尽量与原始 `html/` 目录中的页面一致。
3. React 工程只负责接入路由、状态和数据驱动，不随意增加新的提示文案或新的版式结构。

因此，老师或同学在对比原始页面和重构后的页面时，可以明显看出界面风格基本保持一致，但底层实现方式已经从静态页面升级为 React。

## 4. 数据组织方式

本项目把主要展示数据集中放入 `src/data/Data.json`，包括：

1. 站点基础信息与导航项。
2. 主页轮播图数据。
3. 书籍列表与书籍详情数据。
4. 购物车数据。
5. 订单数据。
6. 个人信息页数据。
7. 登录页文案数据。

由于 `Data.json` 不适合直接存放图片模块引用，因此额外设计了两个文件：

1. `assets.js`
   负责统一导入 logo、书籍封面图和 banner 图。
2. `library.js`
   负责把 JSON 数据和静态资源进行组装，同时补充派生字段，例如：
   `priceText`
   `cover`
   `amountText`

这样做以后，页面组件只关心“如何展示”，而不需要在组件内部处理图片映射和数据格式化问题。

## 5. React 构件化设计原则

### 5.1 公共布局构件

`AppLayout.jsx` 负责渲染导航栏、主内容区和页脚。所有页面共用这一套布局，避免在每个页面中重复编写同样的结构。

### 5.2 可复用展示构件

根据原始页面中重复出现的结构，提取出以下函数组件：

1. `BannerCarousel`
   封装主页轮播图逻辑。
2. `BookCard`
   封装书籍卡片，在书籍列表中重复使用。
3. `SearchBar`
   封装主页顶部搜索区域。
4. `CartRow`
   封装购物车单行条目。
5. `OrderRow`
   封装订单单行条目。
6. `ProfileCard`
   封装个人中心的通用卡片容器。
7. `AddressItem`
   封装地址列表单项。
8. `SettingsItem`
   封装账户设置列表单项。

这些组件职责单一，结构稳定，既符合 React 构件化思想，也方便后续继续扩展功能。

### 5.3 页面组件只负责组合

例如：

1. `BookListPage.jsx` 负责组织书籍页的标题、搜索框、轮播图和书籍网格。
2. `BookDetailPage.jsx` 负责组织封面区、基础信息区和简介区。
3. `ProfilePage.jsx` 负责组合用户信息卡、地址卡和账户设置卡。

也就是说，页面组件负责“组合构件”，通用构件负责“展示细节”。

## 6. 路由设计

本项目使用 React Router 管理页面跳转，主要路由如下：

1. `/`
   主页，也就是书籍列表页。
2. `/books/:bookId`
   书籍详情页。
3. `/cart`
   购物车页。
4. `/orders`
   订单页。
5. `/profile`
   个人信息页。
6. `/login`
   登录页。

这样一来，用户启动项目后就可以像访问真正的单页应用一样，在这些页面之间完成切换。

## 7. 页面之间的数据关联方式

题目要求页面之间通过 `Prop` 和 `State` 传递数据，本项目的实现方式如下：

### 7.1 通过 Props 传递

1. `App.jsx` 从 `library.js` 中读取统一数据。
2. `App.jsx` 再把页面所需的数据通过 `props` 传给各个页面组件。
3. 页面组件继续把局部数据通过 `props` 传给子组件，例如把 `book` 传给 `BookCard`。

这体现了 React 中常见的“自上而下”数据流。

### 7.2 通过 State 传递

在主页点击书籍卡片时，使用 React Router 的 `Link`：

```jsx
<Link to={`/books/${book.id}`} state={{ book }}>
```

这里把当前点击的书籍对象放进了路由 `state` 中。

在详情页对应的路由组件中：

1. 优先从 `location.state.book` 读取当前书籍。
2. 如果用户直接刷新详情页导致 `state` 丢失，再根据 URL 中的 `bookId` 从书籍数组里查找对应数据。

这种设计既满足了题目中“通过 State 传递数据”的要求，也保证了刷新页面时详情页仍然可以正常展示。

## 8. 为什么这样改更适合作业要求

这次重构比直接保留静态 HTML 更符合课程目标，原因有三点：

1. 页面跳转已经由 React Router 管理，不再依赖多个彼此分离的静态页面。
2. 页面数据已经统一收口到 `Data.json`，后续维护成本更低。
3. 页面已经拆成多个函数组件，具备了后续继续迭代和扩展的基础。

同时，这种做法又没有偏离原始界面，能够体现“在已有页面基础上进行 React 化改造”的思路。

## 9. 总结

本项目最终实现的是一种“保留原页面表现、升级底层实现方式”的重构方案：

1. 页面视觉和结构尽量保持原样。
2. 页面跳转改为 React Router 管理。
3. 页面数据集中到 `Data.json`。
4. 页面结构拆分为多个函数组件。

因此，它既满足了作业对 React、React Router、Props、State、统一数据源和构件化开发的要求，也保留了原有页面的整体风格与内容。
